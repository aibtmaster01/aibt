# 로직: 문제 생성 (AI-Gen Universal Framework)

> LLM 기반 객관식 문항 자동 생성·검수 파이프라인. 백엔드 `backend/Contents/` 기준.

---

## 1. 시스템 개요

LLM의 3대 한계를 수학적/결정론적 알고리즘으로 원천 차단하는 문항 생성 시스템.

| 한계 | 대응 방식 |
|------|-----------|
| **환각(Hallucination)** | Pydantic CoT 강제화, Python Native Shuffling, 3-Tier Auditor |
| **포맷 붕괴** | JSON 스키마·Negative Constraints, Strict Index Validation |
| **논리적 불일치** | 선지 타입 → 꼬리말 → 오답 기획 순차 통제, 부정 논리·해설 일치 검증 |

---

## 2. 파이프라인 단계

### Step 1. Blueprint Extractor (DNA 추출)

| 기술 | 구현 파일 | 상세 |
|------|-----------|------|
| TOC-Aware Semantic Parsing | `Default/01_BluePrintExtracter.py` | PyMuPDF `get_toc()`로 목차 추출 → 논리적 단원 인식 |
| Map-Reduce 데이터 압축 | `01_BluePrintExtracter.py` | Counter로 hierarchy 빈도수 집계 → 35,000자 이내 압축 |
| 비동기 에러 방어 | `01_BluePrintExtracter.py` | AsyncOpenAI + tenacity(지수 백오프) |

SQLD 전용: `SQLD_raw/SQLD_01_DNAextractor.py` — Zone A/B/C 구간별 차등 스텝, `ALLOWED_TOPICS`·`ALLOWED_TYPES` 제약

### Step 2. Draft Generator (초안 생성)

| 기술 | 구현 파일 | 상세 |
|------|-----------|------|
| Pydantic CoT (사고 과정 강제화) | `Default/02_Generator.py` | `step0_concept_planning` → `step1_option_type` → `step2_ending` → `step3_trap_design` → `final_question_text` |
| Python Native Shuffling | `02_Generator.py` | AI는 `is_correct`만 생성, `random.shuffle()` 후 인덱스 재할당 → 정답 인덱스 오류율 0% |
| 적응형 컨텍스트 엔진 | `02_Generator.py` | `DOMAINS` 비어있으면 이론 모드, 채우면 실무 시나리오 주입 |

**Negative Constraints (지엽 방지)**:
- "작성하시오", "도출하시오" 등 주관식 어미 금지
- 이중 질문 금지, 지시는 지문 맨 마지막 1회만

### Step 3. 3-Tier Auditor (검수·절삭)

| 기술 | 구현 파일 | 상세 |
|------|-----------|------|
| Char-Ngram TF-IDF 중복 제거 | (특허 명세) | 코사인 유사도 85~90%↑ 문항 사전 폐기 |
| 3인 합의제 검수 | `SQLD_raw/SQLD_03_A_Auditor3Way.py` | Logic Master / Style Editor / Difficulty Judge, S/A/B/F 등급 |
| Strict Index Validation | `SQLD_04_MetadataInjector.py` | 정답 텍스트 ↔ 선지 인덱스 매칭, 피드백 키 2차 교차 검증 |
| 계층적 쿼터 절삭 | (특허 명세) | 블루프린트 단원별 60% 필수 할당 → 잔여분 퀄리티 점수 경쟁 |

**3-Way Auditor 체크리스트**:
- Logic: 부정 질문과 정답 매칭, 해설·answer 일치
- Style: HTML 테이블 컬럼/셀 수 일치, 가독성
- Difficulty: 문법 혼용 여부, 난이도 밸런스

### Step 4~5. Metadata Injector + Gap Analyzer

| 기술 | 구현 파일 | 상세 |
|------|-----------|------|
| Heuristic Metadata | `SQLD_04_MetadataInjector.py` | 지문 길이·HTML 유무 기반 `estimated_time_sec`, `difficulty_level`, `trap_score` 산출 |
| Global Gap Analysis | `SQLD_02_B_GapAnalyzer.py` | Topic별 10개 미만, Low/High 각 3개 미만 → `missing_report.json` |
| Sniper Reinforcer | `SQLD_plus01.py` 등 | 결손 단원 타겟 프롬프트로 100% 수량 복구 |

### Step 6~7. Funnel Allocator (퍼널 배정)

| Round | 배정 기준 | 구현 |
|-------|-----------|------|
| **R1 (신뢰 구축)** | 난이도 2~3, trap_score ≤ 2 | `SQLD_05_FinalAssembler.py` |
| **R2 (결제 유도)** | 최신 30% + 함정 70% | `SQLD_05_FinalAssembler.py` |
| **R3 (프리미엄)** | 난이도 3~4, 최신·긴 지문 | `SQLD_05_FinalAssembler.py` |
| **R4+ (Pool)** | 나머지 전체 | `SQLD_05_FinalAssembler.py` |

## 4. 핵심 기술 발명 포인트

| 청구항 | 기술 내용 |
|--------|-----------|
| **인덱스 동기화** | LLM 텍스트 생성 + Python `random.shuffle()` + Set 교차 검증 → 정답 인덱스 오류 0% |
| **검증 및 교정** | AI가 논리·형식 결함을 Boolean으로 판단 강제(CoV), 오류 시 부분 필드 자가 교정 |
| **계층적 서바이벌 절삭** | Blueprint 비율 + AI 평가 점수 동시 연산, 특정 주제 멸종 방지 |
| **휴리스틱 메타데이터** | AI 추론 없이 텍스트/HTML 구조 기반 가중치로 소요시간·난이도 일관 부여 |

## 5. Firestore 문제 구조

프론트엔드에서 사용하는 문제 데이터 경로:

```
certifications/{certCode}/question_pools/{hierarchy}/questions/{docId}
```

주요 필드: `q_id`, `question_text`, `options`, `answer`, `explanation`, `ai_explanation`, `wrong_feedback`, `hierarchy`, `tags`, `subject_number`, `problem_types`, `difficulty_level`, `trap_score`, `estimated_time_sec`
